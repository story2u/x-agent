# 开发流程

## 1. 需求定位

收到需求后先判断类型：

- CLI / TUI 操作面
- agent 生成 / prompt / tool schema
- 模型凭据（OAuth / API key / `.env`）
- 本地 skills（SKILL.md 组织、选择、校验）

然后从 `docs/功能导航.md` 找到对应功能文档。

## 2. 阅读文档

编码前必须阅读：

1. `docs/开发规范.md`
2. `docs/架构文档.md`
3. `docs/开发流程.md`
4. `docs/功能导航.md`
5. 对应 `docs/features/*.md`

涉及业务流程时阅读 `docs/业务流程.md`。涉及请求 / 模型调用 / 凭据数据流时阅读 `docs/数据流程.md`。

## 3. 制定改动范围

明确：

- 要改的功能域。
- 涉及的文件。
- 是否需要更新 prompt / tool schema。
- 是否需要更新 docs。
- 需要跑哪些验证命令。

## 4. 实现

- 小步修改，只碰相关文件。
- 不回退用户未授权的改动。
- 不把图片功能带回 TUI，除非文档先升级产品边界。
- 不把 skill source of truth 迁回任何数据库。

## 5. 验证

```bash
npm run typecheck
npm run build
npm run lint
npm test
```

本地验证模型调用时，先用 `/model` 检查凭据，或运行 `npm run chatgpt:oauth` 获取单行 `OPENAI_CODEX_OAUTH_CREDENTIALS` 写入 `.env`。

## 6. 文档回写

每次开发结束前更新：

- 相关 `features/*.md`
- 必要的 `架构文档.md`
- 必要的 `业务流程.md`
- 必要的 `数据流程.md`
- 必要的 `功能导航.md`

文档回写要写清楚“新增 / 修改了什么、影响哪些模块、如何验证”。

## 7. 提交

提交前：

```bash
npm run build && npm run typecheck && npm run lint && npm test
git status --short
git diff --stat
```

本项目是本地 CLI，无部署后端；验证通过后提交并推送分支即可。
